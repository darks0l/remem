# ReMEM Benchmark Notes — 2026-05-02

Added `benchmarks/needle-context-extension.mjs`, a reproducible synthetic retrieval harness for measuring context extension under fixed-window pressure.

## What it measures

- Creates synthetic memories with unique fact IDs, codenames, and PINs.
- Simulates a fixed context window using only the most recent N entries.
- Samples queries only from facts outside that window.
- Reports fixed-context recall vs ReMEM recall, MRR, latency, store time, and corpus/window token pressure.

## Commands run

```bash
npm run build
node ./benchmarks/needle-context-extension.mjs --memories 2000 --queries 120 --contextEntries 100 --limit 5 --seed 1337 --exactTopics
node ./benchmarks/needle-context-extension.mjs --memories 10000 --queries 250 --contextEntries 100 --limit 5 --seed 1337 --exactTopics
node ./benchmarks/needle-context-extension.mjs --memories 2000 --queries 120 --contextEntries 100 --limit 5 --seed 1337 --layers
node ./benchmarks/needle-context-extension.mjs --memories 10000 --queries 250 --contextEntries 100 --limit 5 --seed 1337 --layers
```

## Results

- Core exact-codename lookup, 2,000 memories / ~71k corpus tokens vs ~3.6k fixed window:
  - fixed-context recall@1: 0%
  - ReMEM recall@1/@5: 100%
  - avg query: 1.29ms, p95: 1.73ms

- Core exact-codename lookup, 10,000 memories / ~358k corpus tokens vs ~3.6k fixed window:
  - fixed-context recall@1: 0%
  - ReMEM recall@1/@5: 100%
  - avg query: 6.19ms, p95: 7.67ms

- Layered semantic memory, natural-language query, 2,000 memories:
  - fixed-context recall@1: 0%
  - ReMEM recall@1/@5: 100%
  - avg query: 2.46ms, p95: 2.91ms

- Layered semantic memory, natural-language query, 10,000 memories:
  - fixed-context recall@1: 0%
  - ReMEM recall@1/@5: 50.4%
  - avg query: 6.04ms, p95: 6.97ms
  - Finding: without embeddings, simple keyword scoring ties heavily on common terms and insertion order dominates at larger corpus size.

## Important finding

The core `query()` path is currently too literal for natural-language benchmark queries without embeddings: it does `content LIKE "%full query%"`, so natural-language questions return no rows unless the exact full query substring exists. Exact codenames work perfectly; layered query is more flexible because it scans/scores entries in memory.

Embedding run was attempted against `http://192.168.68.69:11434` but Ollama was unreachable from this machine at benchmark time (`fetch failed`; `/api/tags` unable to connect). Re-run with `--embeddings` once Ollama is reachable to measure semantic recall.

## Safe public claim shape

> In a synthetic fixed-window stress test, ReMEM recovered facts stored outside a simulated recent context window across a 10,000-memory / ~358k-token corpus with 100% recall for exact-key lookup at ~6ms average query latency. Natural-language layered retrieval hit 100% at 2,000 memories and exposed the expected need for embeddings/better lexical scoring at 10,000 memories.

Do not claim general semantic 100% recall until embedding-backed results are measured.

## Embedding rerun after Ollama endpoint fix — 2026-05-02 21:39 EDT

Verified after gateway restart:

- `openclaw gateway status`: connectivity probe ok, listening on `127.0.0.1:18789`.
- `memory_search`: provider `ollama`, model `nomic-embed-text`.
- Ollama raw `/api/embeddings` latency probe against `http://192.168.68.69:11434`: ~508ms, 768 dimensions.

Commands run:

```bash
node ./benchmarks/needle-context-extension.mjs --memories 50 --queries 20 --contextEntries 10 --limit 5 --seed 1337 --embeddings --embeddingBaseUrl http://192.168.68.69:11434
node ./benchmarks/needle-context-extension.mjs --memories 50 --queries 20 --contextEntries 10 --limit 5 --seed 1337 --exactTopics
node ./benchmarks/needle-context-extension.mjs --memories 50 --queries 20 --contextEntries 10 --limit 5 --seed 1337
```

Results:

- Core natural-language lookup with embeddings, 50 memories / ~1.8k corpus tokens vs ~353 fixed-window tokens:
  - fixed-context recall@1: 0%
  - ReMEM recall@1/@5: 100%
  - MRR: 1.0
  - avg query: 50.29ms, p95: 60.79ms
  - store time: 29.47s (~589ms per memory)

- Core exact-codename lookup without embeddings, 50 memories:
  - fixed-context recall@1: 0%
  - ReMEM recall@1: 95%, recall@5: 100%, MRR: 0.975
  - avg query: 0.54ms, p95: 0.94ms
  - store time: 17.3ms

- Core natural-language lookup without embeddings, 50 memories:
  - fixed-context recall@1: 0%
  - ReMEM recall@1/@5: 0%
  - avg query: 0.34ms

Finding: embeddings fix natural-language recall in the benchmark, but synchronous ingestion is currently the bottleneck. Larger clean runs were abandoned because the harness waits on one Ollama embedding call per stored memory and provides no progress output; at ~0.5s/vector, 2,000 memories implies ~17+ minutes before queries. Layered embedding runs are not clean evidence yet because `storeInLayer()` fire-and-forgets embedding generation and may query before all vectors land.

Next product fix: add awaited/concurrent layer embedding support or benchmark-side precompute/cache/progress controls before publishing larger semantic-recall claims.
